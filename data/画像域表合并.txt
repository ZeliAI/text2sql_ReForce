# text2sql线上链路

# 现状
+ 链路：语义判断——主题域解析——知识库匹配——意图解析——意图替换——意图替换校验——sql生成——sql校验
+ benchmark比较粗，题型单一，反应不了text2sql模型现状
+ text2sql模型用的是裸模，且都是大尺寸，效率和效果都有优化空间
+ 整个workflow比较定制化，依赖人工写规则，迭代和迁移难度大
+ 行业知识库以RAG形式匹配，效果瓶颈
+ 表结构注入形式未知，存在大段scheme注入，导致的信息污染风险
+ 没有可视化工具

## 主题域解析（LLM）

识别属于哪个主题域

```plsql
你是专业的NL数据主题域解析专家，具备投放域、push域、画像域、策略域的专业领域认知和自然语言语义分析能力，擅长精准捕捉用户查询的核心语义，严格区分跨域模糊场景，确保主题域判定唯一、准确。请严格按照以下所有规则，对用户输入的自然语言查询语句进行唯一主题域判定，仅输出判定结果（仅从指定域中选择其一），无任何多余内容、无解释、无补充。

一、四大主题域核心界定（必严格匹配）
投放域：所有与商业化投放、流量投放相关的查询，核心包含：
投放供给、展位、广告位、流量分配
投放效果指标：曝光量、点击量、转化率、PV、UV、CTR（pvctr/uvctr）、CPC、消耗、ROI
投放人群：定向投放、人群包、投放圈选范围
关键词示例：PV、曝光、展位、点击、流量消耗、商业化、广告、投放、转化、任务

画像域：所有与用户画像、用户特征相关的查询，核心包含：
用户标签：系统标签、自定义标签、行为标签、静态属性，权限，等级
用户分群：基于特征/行为/偏好划分的群体
用户属性：性别、年龄、地域、职业、设备、LBS、活跃区域、状态、生命周期、手机号、身份证、邮箱
用户行为特征：登录、支付行为、打车、持仓、领券、核销、浏览、购买、留存、近N天行为、用户开通、开启、允许、授权
用户偏好/意图：喜欢、经常、最可能、偏好、兴趣、需求
关键词示例：标签、分群、性别、年龄、LBS、活跃区域、打车、支付距今、持仓、登录、领券、核销、近N天行为、经常、喜欢、开通

push域：所有与推送触达相关的查询，核心包含：
push内容（文案、标题、素材）
push发送配置：时间、频次、渠道（APP弹窗、短信、站内信）
push效果指标：发送量、成功量、失败量、触达率、打开率、退订率
push人群：定向推送用户、推送分群
关键词示例：触达、送达、打开、退订、发送



策略域：所有与业务策略、规则引擎、场景化运营逻辑相关的查询，核心包含：
交通场景（如打车调度、路线推荐）、缴费场景（水电煤、话费充值）、EM+场景（权益发放、任务体系）
策略规则：触发条件、优先级、组合逻辑、A/B测试配置
营销/推荐/服务策略：非投放类、非push类、非纯画像的策略设计
不涉及具体投放效果、推送执行或用户标签构建，而是聚焦"怎么做""按什么规则执行"
关键词示例：缴费场景/缴费行业/交通场景/交通行业

二、核心判定规则（必严格执行）

唯一性原则：无论查询包含多少域相关词汇，仅判定为最贴合核心语义目的的一个主题域。拒绝跨域、模糊或多选。

语义目的优先原则：
若目的是描述、筛选或分析用户特征，权限，等级等（如"高活跃用户""正常状态用户""push推送权限""转账权限"），归为画像域
若目的是评估或优化推送效果/配置（如"昨日push打开率"），归为push域
若目的是衡量广告/流量/商业化效果（如"某展位CTR"），归为投放域
若涉及行业或场景化策略（如"缴费行业/交通行业"），归为策略域
关键词上下文敏感原则：

"广告点击率" → 投放域
"push点击率" → push域
"经常点击push的用户" → 画像域（因用于刻画用户行为特征）
"策略"需看修饰对象：
"投放策略" → 投放域
"push策略" → push域
"分群策略" → 画像域
"交通策略" → 策略域
"权限，授权，等级" → 画像域
默认兜底原则：

若查询与四大域均无关，仍强制从四者中选一，默认为投放域
三、最终输出要求

仅输出一个结果，格式为：XX域（如：画像域）
不得包含任何标点、空格、解释、换行或额外字符
严格遵循上述界定与规则，不自行扩展或主观推断

{{input}}
```

## 意图解析（LLM）


进行关键字段映射，缩小下游探索范围

```plsql
你是一个企业级自助取数平台的意图解析器，请从用户自然语言以及知识库中的内容提取结构化要素。**仅做关键词匹配与字段填充，禁止推理、解释或生成额外内容**。

### 输入
用户查询：${{input}}
知识库内容:
{{strategy_info}}
{{push_info}}
{{delivery_info}}
{{user_profile1}}


昨天日期：
 {{TEMPLATE_RESULT}}


 解析流程与规则
 总则：按顺序执行以下1-4模块。模块3（filters）的解析结果将直接影响模块4（time_range）的执行逻辑。

 意图解析状态判定：
• 若能按规则完成各模块的字段提取（即使部分字段为空），且无语义冲突或格式错误 → parse_status = "解析正常"；
• 若出现以下任一情况 → parse_status = "解析异常"：
  - 查询完全无法识别任何有效指标、维度、过滤条件或时间表达；
  - 表述模糊、矛盾、语法混乱或包含系统不支持的语义结构；
  - 时间表达非法、冲突或无法转换为标准 YYYYMMDD 格式；
  - 明确要求某字段但该字段不在知识库且无默认兜底策略；
  - 比较类查询（如"环比增长"）未提供可识别的基准指标。




1. metrics（指标）
• 目标：识别用户查询中明确提到的业务指标。
• 规则：
  - 提取用户输入中明确出现的指标词，例如："用户数""曝光pv""订单金额""点击率""转化率"等。
  - 若查询中包含"上升""下降""增长""提升""环比""同比""对比"等比较性词汇，则必须同时包含原始指标和"提升率"。例如："曝光相比元旦上升了多少" → ["曝光pv", "提升率"]。
  - 支持特殊映射：如"流量消耗"映射为"曝光pv"，"用户"映射为"用户数"，当查询本质是询问某类人群的规模（如"有多少""数量""规模""人数"或直接以人群描述结尾）时，默认指标为"用户数"。
• 输出：数组。未识别到指标时返回空数组 []。

2. dimensions（维度）
• 目标：识别用于数据分组的属性例如：展位名称、用户等级、渠道、投放任务、设备类型、供给、日期、天、周、月等。
• 规则：
(1) 当用户使用"按...统计""...的趋势""...的分布""每天/每周/每月""top N 的 X"等显式分组语义时，提取其中的名词作为维度；
(2) 当用户使用疑问结构如"哪些【商品】""哪些【渠道】"时，方括号内的名词即为维度；
(3) 若输入匹配知识库中的画像域特殊字段（如"最近7天活跃区域"），且该字段用于分组（如"按最近7天活跃区域统计"），则将该完整字段名加入 dimensions；
(4) 知识库特殊字段：若输入文本完全匹配知识库中的画像域字段（如"最近7天活跃区域"），且上下文表明用于分组，则将该完整字段名加入维度。
• 重要标记：检查提取出的维度中是否包含以下任何时间粒度关键词："日期"、"天"、"日"、"周"、"月"、"年"。
 ∘ 如果包含，则设置标记："time_expressed_in_dimension": true。此标记将直接影响模块4。
• 输出：数组。未识别到时返回空数组 []。

3. filters（过滤条件）
• 目标：提取对数据进行筛选的属性约束。
• 规则：
  ✅ 显式提及：如"用户等级=VIP""渠道=APP""省份=北京"，直接映射为标准字段；
  ✅ 隐式映射：根据上下文常识推断， 例如"喜欢xx/最可能xx/最有可能xx/经常xx" → {"字段名": "xx偏好", "操作符": "=", "值": "高"}，"经常看视频" → {"字段名": "视频偏好", "操作符": "=", "值": "高"}；
✅ 隐式映射：根据上下文常识推断，例如 有过/参与过/xx过 表名曾经进行过某些业务动作等，在明确目标字段的前提下，根据知识库的枚举值样例，如果是数值类枚举值，则操作符为和比较值为>=1，若为字符串数组或者分隔符字符串，则为like具体筛选值；
  ✅ 返回格式为 JSON 列表，每个元素为对象：{"字段名": "标准字段名", "操作符": "=", "值": "筛选值"}，支持操作符包括 =、!=、>、<、>=、<=、IN、like；
  ✅ 多个独立条件必须全部提取，不得合并或省略；
  ✅ 若为模糊描述（如"高价值用户""优质客户"），统一标注为 {"字段名": "未知", "操作符": "=", "值": "高价值用户"}；
• 重要标记：检查提取出的 filters 中，是否有任何"字段名"包含以下时间窗口后缀：_r7d, _last7d, _recent30d, _30d,_7d。
 ∘ 如果包含，则设置标记："time_embedded_in_filter": true。此标记将直接影响模块4。
• 输出：JSON对象数组。无过滤条件时返回空数组 []。

4. time_range（时间范围）
第一步：前置判断（跳过条件）
如果满足以下任一条件，则立即采用"兜底规则"，无需执行后续解析：
1. 来自模块2的标记 time_expressed_in_dimension 为 true。
2. 来自模块3的标记 time_embedded_in_filter 为 true。
3. 用户查询中完全没有任何形式的时间表述（如"最近"、"昨天"、"2025年"等）。
若触发跳过条件，则直接输出以下"兜底规则"结果： { "raw_time_text": "无", "time_range_type": "最近 N 天", "target_time_range": {"type": "最近N天", "N": 1, "start_date": "{{TEMPLATE_RESULT}}", "end_date": "{{TEMPLATE_RESULT}}"}, "base_time_range": {}, "time_compare_type": "无" }
第二步：正常解析（仅当不满足上述跳过条件时执行）
  ● 所有日期必须使用 YYYYMMDD 格式，禁止使用"2026-02-01""2月1日"等其他形式；
  ● {{TEMPLATE_RESULT}} 表示昨天的日期，{{TODAY}} 表示今天的日期（仅用于"今天"及对比场景）；
  ● 动态时间范围（如"最近N天""本周""本月"）的起止日期必须严格基于 {{TEMPLATE_RESULT}} 推导，不得使用用户输入中的具体日期；仅"自定义日期"类型允许使用用户提供的日期；
  ● 必须完整输出以下五个字段：
    (1) raw_time_text：完整保留用户原始时间表述，若无时间词则填 "无"；
    (2) time_range_type：枚举值之一（最近 N 天、昨天、今天、本周、本月、上月、自定义日期、节假日、时间对比）；若含"同比""环比""对比""较上周""较昨日"等比较词汇，强制设为 "时间对比"；
    (3) target_time_range：主时间范围，结构为 {"type": "时间类型", "N": 正整数（仅最近N天使用）, "start_date": "YYYYMMDD", "end_date": "YYYYMMDD"}；
    (4) base_time_range：对比基准范围，非对比场景填 {}，对比场景结构同 target_time_range；
    (5) time_compare_type：枚举值（周同比 / 日环比 / 月同比 / 自定义对比 / 无），仅在 time_range_type 为 "时间对比" 时赋有效值，否则为 "无"。
  ● 核心解析规则：
  (1) 基础时间类型：
  - "最近 N 天"：N 可为中文数字（如"七"→7,"三十"→30），end_date = {{TEMPLATE_RESULT}}，start_date = {{TEMPLATE_RESULT}} 减去 (N - 1) 天；
  - "昨天"：start_date = end_date = {{TEMPLATE_RESULT}}；
  - "今天"：start_date = end_date = {{TODAY}}；
  - "本周"：end_date = {{TEMPLATE_RESULT}}，start_date 为本周一；
  - "本月"：end_date = {{TEMPLATE_RESULT}}，start_date 为当月1日；
  - "上月"：start_date 为上月1日，end_date 为上月最后一天；
  - "自定义日期"：仅当用户明确提供具体日期（如"20251201 到 20251207"）时使用，支持"到""至""-""~"等分隔符；单日期则 start_date = end_date = 该日期；若缺结束日期，默认 end_date = {{TEMPLATE_RESULT}}；
  - "节假日"：节日名称映射为具体日期（假设有内置节日表）；含"期间"则按默认假期长度生成区间，否则视为单日。
(2) 时间对比场景：
  - 触发词包括"同比""环比""对比""较上周""较昨日"等；
  - time_range_type 强制设为 "时间对比"；
  - target_time_range 按用户指定目标期解析；
  - base_time_range 自动匹配：
    • 周同比：基准为 {{TODAY}} - 7 天（单日）；
    • 日环比：基准为昨天（{{TEMPLATE_RESULT}}）；
    • 月同比：基准为上月同期区间；
    • 自定义对比：按用户输入分别解析基准期；
  - time_compare_type 对应赋值为"周同比""日环比""月同比"或"自定义对比"。
(3) 口语化转换（强制执行）：
- "近 N 天""最近 N 日"等统一识别为 "最近 N 天"；
- 中文数字"三""七""十"转为阿拉伯数字 3、7、10；
- "上月"自动解析为上月1日至最后一天；
- "上周"自动解析为上周周一至上周日。

- 所有字段必须存在，无值按规则兜底（字符串→"无"，数组→[]，对象→{}；
```

     ```python
     from datetime import datetime, timedelta
     
     # 基础转换函数
     def str_to_date(date_str):
         return datetime.strptime(date_str, "%Y%m%d")
     
     def date_to_str(date_obj):
         return date_obj.strftime("%Y%m%d")
     
     # 日环比计算
     def calculate_day_on_day(base_start, base_end):
         base_s = str_to_date(base_start)
         base_e = str_to_date(base_end)
         compare_s = base_s - timedelta(days=1)
         compare_e = base_e - timedelta(days=1)
         return date_to_str(compare_s), date_to_str(compare_e)
     
     # 周同比计算
     def calculate_week_on_year(base_start, base_end):
         base_s = str_to_date(base_start)
         base_e = str_to_date(base_end)
         compare_s = base_s - timedelta(days=7)
         compare_e = base_e - timedelta(days=7)
         return date_to_str(compare_s), date_to_str(compare_e)
     
     # 月同比计算（处理月末边界）
     def calculate_month_on_year(base_start, base_end):
         from dateutil.relativedelta import relativedelta
         base_s = str_to_date(base_start)
         base_e = str_to_date(base_end)
         
         # 计算上月同期
         compare_s = base_s - relativedelta(months=1)
         compare_e = base_e - relativedelta(months=1)
         
         # 处理2月等边界情况
         if base_s.month != compare_s.month + 1:

             compare_e = compare_e.replace(day=1) - timedelta(days=1)
             
         return date_to_str(compare_s), date_to_str(compare_e)
     ```

# 后续优化方案
## 意图理解层
+ **语义澄清机制**：当用户输入模糊时（如"任务ID"），主动引导明确实体类型（投放任务？策略任务？）；
+ **多主题域联合识别**：支持跨域条件组合（如"曝光 + push"）；
+ **时间-行为解耦**：区分"时间范围"与"行为中的时间属性"（如"最近7天登录" vs "注册距今7天"）。

## 知识召回层
+ **意图驱动召回**：将意图解析出的关键词（如"eM+"、"港漂"）作为召回增强信号；
+ **枚举值标准化**：建立业务术语 ↔ 标准字段/值的映射词典（如"年轻人" → `user_age_group IN ('2','3','4')`）。

## SQL 生成层
+ **禁止伪造**：无匹配字段/值时，返回"目前用户查询无法满足"；
+ **支持高级语法**：
    - `LIKE`（前缀匹配，如"86开头"）；
    - `GROUP BY + HAVING`（如"活跃≥3个场景的用户数"）；
    - 安全除法（`NULLIF`）；
+ **性能兜底**：自动添加 `dt = max_pt(...)` 或分区裁剪。

## 系统稳定性
+ **超时控制**：单节点执行超时熔断；
+ **Token 限制**：对长上下文进行截断或分片处理；
+ **错误分类上报**：区分"知识缺失"、"语义模糊"、"生成错误"等根因。

## 总结与建议
**短期目标**：聚焦单主题域，打造高可靠 NL2SQL 能力；  
**长期目标**：通过语义澄清 + 知识增强 + 规则引擎，突破 80% 准确率天花板。



当前主要挑战包括：

+ 用户表述模糊、口语化严重；
+ 多主题域、多条件混合查询；
+ 时间语义嵌套复杂；
+ 知识库覆盖不全导致字段/值伪造。

## 二、表关联关系
```plain
adm_asap_base_user_label_dd (user_id, dt)
adm_asap_algo_user_label_dd (user_id, dt)
adm_asap_pay_user_label_dd (user_id, dt)
adm_asap_other_action_user_label_dd (user_id, dt)
```

**关联方式：** 4张表均为平级关系，通过 `user_id` + `dt` 进行 JOIN，均为 1:1。  
**选表原则：查询维度在哪张表，直接用那张表统计，无需强制关联 base 表。** 仅在需要跨表字段组合时才做 JOIN。

## 三、选表规则（NL2SQL 核心）
| 查询意图 | 使用表 |
| --- | --- |
| 用户状态/年龄/性别/认证/地区/生命周期/价值/风控 | adm_asap_base_user_label_dd |
| 权益偏好/展位偏好/业务偏好/品类偏好/eM+潜客/基金潜客/回流沉默预警 | adm_asap_algo_user_label_dd |
| 交易频次/交易金额/交易场景/支付渠道/退款/用券 | adm_asap_pay_user_label_dd |
| 活跃度/登录次数/扫码/转账/页面浏览/搜索/push互动 | adm_asap_other_action_user_label_dd |
| 需要跨表字段组合（如：eM+高潜用户的年龄分布） | 各表子查询过滤后 JOIN |


```plain
选表判断流程：
查询所需字段全在一张表？
  是 → 直接单表查询，无需 JOIN
  否 → 各表子查询先过滤 dt + 业务条件，再按 user_id + dt JOIN
```

## 七、常用查询场景 SQL 示例
### 7.1 查询高价值用户画像分布（单表 base）
```sql
-- 查询维度（年龄/性别/港漂）均在 base 表，直接单表查
SELECT
    user_age_group,
    user_gender,
    is_hk_drifter,
    COUNT(DISTINCT user_id) AS user_cnt
FROM antsg_asap.adm_asap_base_user_label_dd
WHERE dt = '${bizdate}'
  AND user_value LIKE '1%'
  AND is_promo_black_flag = '0'
GROUP BY user_age_group, user_gender, is_hk_drifter
ORDER BY user_cnt DESC
```

### 7.2 查询eM+各潜力等级用户数（单表 algo）
```sql
-- eM+ 相关字段全在 algo 表，直接单表统计，无需关联 base
SELECT
    em_potential_segment,
    COUNT(DISTINCT user_id) AS user_cnt
FROM anthk_sg.adm_asap_algo_user_label_dd
WHERE dt = '${bizdate}'
GROUP BY em_potential_segment
ORDER BY user_cnt DESC
```

### 7.3 查询各交易场景用户数及笔均金额（单表 pay）
```sql
-- 交易维度全在 pay 表，直接单表统计
SELECT
    primary_trx_scene_r30d,
    COUNT(DISTINCT user_id) AS user_cnt,
    AVG(avg_trx_amt_r30d) AS avg_trx_amt
FROM antsg_asap.adm_asap_pay_user_label_dd
WHERE dt = '${bizdate}'
GROUP BY primary_trx_scene_r30d
ORDER BY user_cnt DESC
```

### 7.4 查询沉默用户的回流潜力分布（需 JOIN）
```sql
-- 沉默条件在 other_action，回流潜力字段在 algo，跨表才需要 JOIN
SELECT
    b.reactivation_level,
    COUNT(DISTINCT a.user_id) AS user_cnt,
    AVG(a.last_active_days) AS avg_silent_days
FROM (
    -- other_action：过滤日期 + 沉默用户
    SELECT user_id, last_active_days
    FROM antsg_asap.adm_asap_other_action_user_label_dd
    WHERE dt = '${bizdate}'
      AND last_active_days > 30
) a
JOIN (
    -- algo：过滤日期
    SELECT user_id, reactivation_level
    FROM anthk_sg.adm_asap_algo_user_label_dd
    WHERE dt = '${bizdate}'
) b ON a.user_id = b.user_id
GROUP BY b.reactivation_level
ORDER BY user_cnt DESC
```

### 7.5 查询eM+高潜用户的年龄性别分布（需 JOIN）
```sql
-- eM+高潜 在 algo，年龄/性别 在 base，跨表才 JOIN
SELECT
    a.user_age_group,
    a.user_gender,
    COUNT(DISTINCT a.user_id) AS em_potential_cnt
FROM (
    -- base：过滤日期
    SELECT user_id, user_age_group, user_gender
    FROM antsg_asap.adm_asap_base_user_label_dd
    WHERE dt = '${bizdate}'
) a
JOIN (
    -- algo：过滤日期 + eM+高潜
    SELECT user_id
    FROM anthk_sg.adm_asap_algo_user_label_dd
    WHERE dt = '${bizdate}'
      AND em_potential_segment = '高潜'
) b ON a.user_id = b.user_id
GROUP BY a.user_age_group, a.user_gender
```

## 八、查询注意事项
1. **分区字段必填**：所有表均为 `dt` 分区，格式 `yyyyMMdd`，查询时必须指定
2. **优先单表**：查询维度在哪张表就直接用那张表，无需强制关联其他表
3. **多表 JOIN 必须先过滤**：确认跨表字段需要 JOIN 时，各子查询内先 `WHERE dt + 业务条件` 再 JOIN，避免全表扫描
4. **用户去重**：用户数统计使用 `COUNT(DISTINCT user_id)`
5. **黑名单过滤**：营销场景需加 `AND is_promo_black_flag = '0' AND is_black_list = '0'`（字段在 base 表）
6. **偏好等级通用**：所有 `*_prefer_level` 字段值域均为 `'high'`, `'medium'`, `'low'`
7. **时间窗口选择**：交易/活跃类指标有 r7d（近7天）和 r30d（近30天）两种，根据需求选择
8. **潜客rank字段**：所有 `*_rank` 字段数值越小排名越高（rank=1 为最高优先级）

## 四、常用查询场景
### 4.1 查询用户基础属性分布
```sql
-- 查询各年龄段用户分布
SELECT 
    user_age_group,
    COUNT(DISTINCT user_id) AS user_cnt
FROM antsg_asap.adm_asap_base_user_label_dd
WHERE dt = '${bizdate}'
  AND user_status = 'T'
GROUP BY user_age_group
ORDER BY user_cnt DESC;
```

### 4.2 查询港漂用户画像
```sql
-- 查询港漂用户的性别和年龄分布
SELECT 
    user_gender,
    user_age_group,
    COUNT(DISTINCT user_id) AS user_cnt
FROM antsg_asap.adm_asap_base_user_label_dd
WHERE dt = '${bizdate}'
  AND is_hk_drifter = 'CX'
  AND user_status = 'T'
GROUP BY user_gender, user_age_group
ORDER BY user_cnt DESC;
```

### 4.3 查询高价值用户特征
```sql
-- 查询高价值用户的生命周期分布
SELECT 
    life_cycle,
    COUNT(DISTINCT user_id) AS user_cnt
FROM antsg_asap.adm_asap_base_user_label_dd
WHERE dt = '${bizdate}'
  AND user_value LIKE '1%'
  AND is_promo_black_flag = '0'
  AND is_black_list = '0'
GROUP BY life_cycle
ORDER BY user_cnt DESC;
```

### 4.4 查询KYC认证情况
```sql
-- 查询各KYC等级的用户分布
SELECT 
    kyc_level,
    COUNT(DISTINCT user_id) AS user_cnt
FROM antsg_asap.adm_asap_base_user_label_dd
WHERE dt = '${bizdate}'
  AND user_status = 'T'
GROUP BY kyc_level
ORDER BY user_cnt DESC;
```

### 4.5 查询用户地理位置分布
```sql
-- 查询近30天活跃区域分布
SELECT 
    active_region_r30d,
    COUNT(DISTINCT user_id) AS user_cnt
FROM antsg_asap.adm_asap_base_user_label_dd
WHERE dt = '${bizdate}'
  AND active_region_r30d IS NOT NULL
GROUP BY active_region_r30d
ORDER BY user_cnt DESC;
```

## 五、注意事项
1. **分区字段必填**：所有查询必须指定 `dt` 分区，格式为 `yyyyMMdd`
2. **用户状态过滤**：统计活跃用户数时，建议添加 `user_status = 'T'` 过滤正常用户
3. **营销黑名单过滤**：营销场景需排除黑名单用户，添加条件：

```sql
AND is_promo_black_flag = '0' 
AND is_black_list = '0'
```

4. **年龄计算**：`user_age = -1` 表示未获取到年龄，统计平均年龄时需排除
5. **生命周期字段说明**：
    - '1': 新手期用户
    - '2': 成长期用户  
    - '3': 成熟期用户
    - '4': 衰退期用户
    - '5': 沉默期用户
    - '6': 流失期用户
    - '61': 流失期用户_非AAU
6. **用户价值字段说明**：以'1'开头表示高价值用户，'8'开头表示低价值用户
7. **活跃区域字段**：包含内地、香港、日本、澳门、泰国、菲律宾、台湾、韩国、英国、马来西亚、新加坡、越南、印度尼西亚、缅甸、柬埔寨等
8. **风控评分字段**：
    - `pl_apply_predict_score`: 标准用户信贷评分，默认9999
    - `pl_apply_predict_score_ph_maid`: 菲佣用户信贷评分，默认9999
    - `pl_apply_user_rank`: 10分制评分，无得分为9999
    - `ploan_riskseg`: PloanA卡评级，范围0-9，默认999
    - `mcard_riskseg`: M卡评级，范围0-9，默认999

## 四、常用查询场景
### 4.1 查询高偏好用户分布
```sql
-- 查询各权益偏好的高偏好用户数
SELECT 
    pts_prefer_level,
    COUNT(DISTINCT user_id) AS user_cnt
FROM anthk_sg.adm_asap_algo_user_label_dd
WHERE dt = '${bizdate}'
  AND pts_prefer_level = 'high'
GROUP BY pts_prefer_level;
```

### 4.2 查询eM+潜客分层
```sql
-- 查询eM+各潜力等级用户数
SELECT 
    em_potential_segment,
    COUNT(DISTINCT user_id) AS user_cnt
FROM anthk_sg.adm_asap_algo_user_label_dd
WHERE dt = '${bizdate}'
GROUP BY em_potential_segment
ORDER BY user_cnt DESC;
```

### 4.3 查询基金潜客
```sql
-- 查询基金高潜用户
SELECT 
    fund_potential_segment,
    COUNT(DISTINCT user_id) AS user_cnt
FROM anthk_sg.adm_asap_algo_user_label_dd
WHERE dt = '${bizdate}'
GROUP BY fund_potential_segment
ORDER BY user_cnt DESC;
```

### 4.4 查询用户回流潜力
```sql
-- 查询各回流潜力等级用户数
SELECT 
    reactivation_level,
    COUNT(DISTINCT user_id) AS user_cnt
FROM anthk_sg.adm_asap_algo_user_label_dd
WHERE dt = '${bizdate}'
GROUP BY reactivation_level
ORDER BY user_cnt DESC;
```

### 4.5 查询沉默风险用户
```sql
-- 查询沉默风险等级分布
SELECT 
    silence_risk_level,
    COUNT(DISTINCT user_id) AS user_cnt
FROM anthk_sg.adm_asap_algo_user_label_dd
WHERE dt = '${bizdate}'
GROUP BY silence_risk_level
ORDER BY user_cnt DESC;
```

### 4.6 查询eM+高潜用户Top N
```sql
-- 查询eM+潜客排名最高的1000名用户
SELECT 
    user_id,
    em_potential_rank,
    em_potential_segment
FROM anthk_sg.adm_asap_algo_user_label_dd
WHERE dt = '${bizdate}'
  AND em_potential_segment = '高潜'
ORDER BY em_potential_rank ASC
LIMIT 1000;
```

## 五、注意事项
1. **分区字段必填**：所有查询必须指定 `dt` 分区，格式为 `yyyyMMdd`
2. **偏好等级字段通用说明**：所有 `*_prefer_level` 字段值域均为 `'high'`(高)、`'medium'`(中)、`'low'`(低)
3. **潜客分层字段说明**：
    - `em_potential_segment`: eM+开通预测（高潜、中潜、低潜）
    - `fund_potential_segment`: 基金开通预测（高潜、中潜、低潜）
    - `em_transfer_in_label`: eM+入金预测（高入金、中入金、低入金）
4. **Rank字段说明**：所有 `*_rank` 字段数值越小排名越高（rank=1 为最高优先级）
5. **eM+用户价值**：'1'表示高价值、'2'表示中价值、'3'表示低价值
6. **营销敏感度**：分为'高营销敏感度'和'低营销敏感度'
7. **时段偏好字段**：r00h-r23h 表示24小时制的各时段偏好等级
8. **关联使用**：该表需与 `adm_asap_base_user_label_dd` 通过 `user_id` + `dt` 进行关联使用

## 五、注意事项
1. **分区字段必填**：所有查询必须指定 `dt` 分区，格式为 `yyyyMMdd`
2. **布尔标志字段**：以 `is_` 开头的字段，'1' 表示是，'0' 表示否
3. **默认值说明**：
    - 交易笔数/金额类字段默认值为 0
    - 天数类字段无记录时默认值为 -1 或 9999
    - 字符串类字段默认值为 null
4. **时间窗口字段**：
    - r1d/r7d/r15d/r30d/r60d/r90d/r365d 表示不同时间窗口的统计
    - his 表示历史至今的累计统计
5. **支付活跃分级**：pay_level_r30d 字段将用户分为 G1-G9 九个等级，从高频高金额到无支付活跃
6. **数据更新**：该表按日更新，dt 分区为数据日期，通常 T+1 更新
7. **关联使用**：该表需与 `adm_asap_base_user_label_dd` 通过 `user_id` + `dt` 进行关联使用
